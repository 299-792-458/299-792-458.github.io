# 오픈소스의 미래

Open Source Summit Korea 2025에서는 화려한 비전보다는, 실제로 돌아가는 시스템과 워크플로를 많이 볼 수 있었다. 세션에서는 AI와 오픈소스가 만나는 지점을 말할 때도, 데이터 파이프라인 구조도와 라이선스 정책, 조직 내 적용 사례가 나란히 등장했다. 커뮤니티 부스에서도 프로젝트 "소개 슬라이드"보다, 실제로 운영 중인 저장소 구조와 자동화 스크립트를 먼저 보여 주는 분위기였다.

![오픈소스 서밋 현장의 포스터와 관객 사진](/log/images/opensource.webp)

발표자들은 "오픈소스가 AI를 이끈다"는 문장을 슬로건이 아니라 운영 전략으로 다뤘다. 대형 모델을 그대로 쓰기보다, 내부 도메인에 맞춰 포크한 레포를 어떻게 관리하는지, 데이터·모델·코드를 분리해서 버전 관리하는지 같은 이야기들이 나왔다. API 키와 접근 권한을 조직 구조와 연결해 설계하는 사례도 인상 깊었다.

<pre style="color:#5AA7FF;font-family:var(--font-mono,monospace);background:none;border:none;padding:0;margin:1em 0">
"코드보다 먼저 설계해야 하는 것은, 누가 무엇을 어떻게 유지보수할지에 대한 책임 구조다."
</pre>

워크숍 자리에서는 "왜 이 프로젝트를 계속 유지하느냐"는 질문이 반복됐다. 누군가는 이슈 템플릿과 라벨링 규칙을 공개해, 신규 기여자가 어디서부터 시작할 수 있는지 보여 줬다. 또 다른 팀은 릴리스 노트를 자동 생성하는 파이프라인을 설명하며, 유지보수자의 시간을 어떻게 아끼는지 공유했다. 이런 이야기를 들으며, 나 역시 집에 돌아오는 길에 내 저장소의 브랜치 전략과 라벨 세트를 다시 그려 보았다.

앞으로는 프로젝트 README에 단순한 사용법을 적는 데서 그치지 않고, 어떤 환경에서 테스트했는지, 어떤 규모까지를 공식 지원 범위로 보는지까지 명시하려 한다. CONTRIBUTING 문서에는 이슈 분류 기준과 리뷰 SLA, 릴리스 주기 등을 구체적으로 적어 둘 생각이다. 우선 내가 관리하는 레포부터 이 기준을 적용해 보고, 문제가 없으면 팀 템플릿으로 승격시킬 계획이다.
